Ralph session: wavefront PT race fix + feature parity
Started: 2026-05-10

Diagnosis:
- wgpu 29: queue.write_buffer accumulates writes to a "pre-submit" stream applied BEFORE encoder commands at submit time. Per-tile writes to single dims_buf/count_buf collapse to last-tile values. Result: only the last tile renders correctly, others get garbage.
- spectral.rs is a stub that forces pt_wavefront=false → tests via spectral=on actually exercise megakernel.
- ReSTIR/PathGuide/Adaptive force-disabled in tile mode as a workaround.

Architecture:
- pt-wavefront crate: pipeline.rs (single-slot dims_buf/count_buf), wavefront/{raygen,intersect,shade,count_swap,finalize}.wgsl shaders.
- pt-megakernel crate: compute.rs::dispatch_wavefront drives the wavefront tile loop and bind-group rebuild via rebuild_wavefront_bind_groups.
- render-3d crate: pt/{wavefront.rs, spectral.rs, megakernel/} backend dispatchers + per-frame ShadeParams/PtCameraUniform writes.

Plan:
US-001: Replace single-slot dims_buf/count_buf in pt-wavefront with N-slot dynamic-offset persistent buffers.
US-002: dispatch_wavefront uses prepare_tiles + reset_tile_count + dynamic offsets, no per-tile queue.write_buffer.
US-003: Spectral routes through wavefront when pt_wavefront=true; shade.wgsl applies spectral_tint properly.
US-004: ReSTIR/PathGuide/Adaptive per-tile param buffers also migrated to dynamic-offset.
US-005: Integration parity test megakernel ≈ wf full ≈ wf tiled.

Codebase notes:
- WfDims = 32 bytes (8 u32). Pad each tile slot to 256 bytes for dynamic offset alignment.
- count_buf currently 16 bytes ([count_in, count_out, _, _]). Pad each tile slot to 256 bytes.
- Tile capacity for FullHD with min tile_size=256: ceil(1920/256) * ceil(1080/256) = 8 * 5 = 40 tiles. Allocate generously (e.g. 64 slots default, grow on demand).
- Storage buffer dynamic offset alignment is also 256 bytes on most adapters.
- min_binding_size enforces "at least N bytes accessible from offset" — set to actual struct size.
